## Context for `c` Paper

### C Language Semantics Relevant to Safe-Rust Translation

#### C's Fundamental Properties (The Problem Statement)

C is the canonical embodiment of maximal runtime deferral. Its type system is a strict *subsystem* of Rust's: every C type has a Rust equivalent, but Rust's type system adds structure (lifetimes, ownership, trait bounds) that C lacks entirely. The Compile-Time Supremacy principle (c2rust_strategy.tex, Principle 1.1) frames the entire translation problem: C→Rust migration is fundamentally a type inference problem — recovering the implicit ownership semantics, lifetime relationships, and aliasing constraints that C code encodes only in programmer intent.

**The Minimal Runtime Axiom:** Every invariant deferred to runtime is an invariant that will eventually be violated. C's raw pointer arithmetic, manual memory management, implicit type coercions, and undefined behavior all represent invariants that exist only in the programmer's mental model, never in the compiler's enforcement regime.

#### C Undefined Behavior (UB) Taxonomy

The C standard's undefined behavior is not merely "implementation-defined" — compilers *optimize based on UB assumptions*. A translated Rust that naively preserves C behavior may preserve behavior the C compiler was silently exploiting for optimization, which was never the programmer's intent. Key UB categories relevant to translation:

1. **Signed integer overflow** — C compilers assume it doesn't happen; LLVM optimizes away overflow checks. Rust's `i32` wraps in debug mode and panics.
2. **Null pointer dereference** — C programs frequently rely on NULL-terminated data structures; Rust's Option type encodes this explicitly.
3. **Use-after-free / double-free** — Rust's ownership eliminates at compile time.
4. **Buffer overread/overwrite** — The defining class of CVEs in libyaml, libexpat, libpng, libxml2. Rust slice indexing panics on OOB; raw pointer arithmetic doesn't.
5. **Type-based aliasing violations (TBAA)** — C allows casting pointers to unrelated types; LLVM exploits the no-alias assumption for optimization. Rust's `&T` / `&mut T` encodes aliasing rules in the type system.
6. **Integer-as-bool pattern** — `int yaml_parser_parse(parser, &event)` returns 0/1; callers often forget to check. Rust's `#[must_use] Result<T, E>` makes this a compile error.
7. **Uninitialized memory reads** — C struct initialization with `memset` vs. field-by-field. Rust requires all fields initialized before use.

#### libyaml Specific C Idioms

**1. Custom string type with manual triple**
```c
// yaml_private.h
typedef struct {
    yaml_char_t *start;   // beginning of allocated buffer
    yaml_char_t *end;     // one past end of allocated buffer
    yaml_char_t *pointer; // current position
} yaml_string_t;
```
Rust equivalent: `Vec<u8>` or a newtype wrapper. The three-pointer idiom encodes capacity, length, and cursor in raw pointer arithmetic. STACK_INIT initializes with 16 elements initially.

**2. Buffer management macro family** (yaml_private.h)
- `STRING_INIT(context, string, size)` — allocates `size` bytes, sets start/end/pointer
- `STRING_EXTEND(context, string)` — reallocs if at capacity (doubles)
- `STRING_DEL(context, string)` — frees, zeros pointers
- `STRING_JOIN(context, string1, string2)` — appends string2 onto string1
- Rust equivalent: `Vec<u8>` with `push`, `extend_from_slice`, automatic drop

**3. Stack macro family** (INITIAL_STACK_SIZE = 16)
- `STACK_INIT(context, stack, type)` — malloc of INITIAL_STACK_SIZE elements
- `STACK_PUSH(context, stack, value)` — push with realloc if full (doubles capacity, calls yaml_stack_extend)
- `STACK_POP(context, stack)` — returns top element, decrements pointer
- `STACK_DEL(context, stack)` — frees memory
- Rust equivalent: `Vec<T>` with `push`, `pop`, `drop`

**4. Queue macro family** (INITIAL_QUEUE_SIZE = 16)
- `QUEUE_INIT(context, queue, type)` — malloc ring buffer
- `ENQUEUE(context, queue, value)` — push to tail, call yaml_queue_extend if needed
- `DEQUEUE(context, queue)` — pop from head
- `QUEUE_DEL(context, queue)` — frees memory
- Rust equivalent: `VecDeque<T>` with `push_back`, `pop_front`, `drop`

**5. Error handling idiom**
```c
int yaml_parser_parse(yaml_parser_t *parser, yaml_event_t *event) {
    // ...
    if (parser->error != YAML_NO_ERROR) return 0;
    // ...
    return 1;
}
```
The error is stored in the parser struct's `error` field alongside `problem` (string) and `problem_mark` (location). Return value 0/1 is ignored by callers surprisingly often. Rust equivalent: `Result<Event, Error>` with `#[must_use]`.

**6. Output parameter pattern**
```c
int yaml_parser_parse(yaml_parser_t *parser, yaml_event_t *event);  // event is output
```
Caller must pre-allocate event on stack, pass pointer. Rust equivalent: return value `Result<Event, Error>` — the output parameter becomes the success case of Result.

**7. Tagged union (yaml_event_t)**
```c
typedef struct yaml_event_s {
    yaml_event_type_t type;  // discriminant tag
    union {
        struct { /* stream start data */ } stream_start;
        struct { /* document start data */ } document_start;
        struct { /* document end data */ } document_end;
        struct { /* alias data */ } alias;
        struct { /* scalar data */ } scalar;
        struct { /* sequence start data */ } sequence_start;
        struct { /* mapping start data */ } mapping_start;
    } data;
    yaml_mark_t start_mark;
    yaml_mark_t end_mark;
} yaml_event_t;
```
Rust equivalent: `enum Event { StreamStart, StreamEnd, DocumentStart {...}, DocumentEnd {...}, Alias {...}, Scalar {...}, SequenceStart {...}, SequenceEnd, MappingStart {...}, MappingEnd }` — the discriminant and data are fused into enum variants.

**8. Custom allocator hooks (yaml.h)**
```c
typedef void *(*yaml_malloc_t)(void *data, size_t size);
typedef void *(*yaml_realloc_t)(void *data, void *ptr, size_t size);
typedef void (*yaml_free_t)(void *data, void *ptr);
```
libyaml supports custom allocators via function pointer fields in the parser/emitter structs. The `data` parameter is `void*` context. Rust equivalent: `bumpalo::Bump` or the global allocator API; for safe Rust, custom allocators implement the `Allocator` trait.

**9. Nullable pointers as optionals**
```c
char *anchor;  // NULL means no anchor
```
Throughout libyaml, fields that may or may not be present are represented as NULL pointers. Rust equivalent: `Option<String>` or `Option<&str>`.

**10. Character predicate macros (scanner.c)**
```c
#define IS_BLANK(string) ((string).pointer[0] == ' ' || (string).pointer[0] == '\t')
#define IS_BREAK(string) ((string).pointer[0] == '\r' || (string).pointer[0] == '\n' || ...)
#define CHECK(string, octet) ((string).pointer[0] == (octet))
#define FORWARD(string) ((string).pointer < (string).end ? (string).pointer++ : ...)
```
Rust equivalent: methods on a custom iterator type or on a `&[u8]` slice with `first()`, indexing, iterator advancement.

**11. Integer-as-bool return convention**
Nearly all libyaml functions return `int` where 1 = success, 0 = failure. This is the pervasive error handling idiom. The error details are stored in the parser/emitter struct. Translating to `Result<T, Error>` both fixes this idiom and makes the error propagation explicit via `?` operator.

**12. setjmp/longjmp (libpng, relevant for Tier 3)**
libpng uses `setjmp`/`longjmp` for error handling — non-local control flow that Rust cannot safely represent. This is the hardest C→Rust translation problem. Solutions involve restructuring error paths, using `panic::catch_unwind`, or representing the error continuation explicitly.

#### libyaml CVE History
libyaml has had multiple buffer overflow and heap overflow CVEs. The YAML parsing specification is complex, with many edge cases around Unicode handling, escape sequences, and deeply nested structures. CVEs have affected:
- Heap overflows in scalar scanning
- Buffer overflows in the reader/encoding detection
- Integer overflows in size calculations

libexpat's CVE history is far more severe: 40+ CVEs including critical remote code execution. CVE-2022-25235 (missing UTF-8 validation in xmltok_impl.c) and CVE-2022-25236 (namespace separator injection) were catastrophic in 2022. CVE-2024-8176 (stack overflow via recursive entity expansion) is more recent. CVE-2025-59375 (dynamic memory allocation vulnerability) was disclosed in 2025. libexpat is used by Python, Apache, Firefox, D-Bus, and scores of critical systems.

#### C Pointer Classification for Translation (from OSG design)
For every C pointer, the Ferrous Bridge pipeline assigns one of:
- **Owning**: malloc/free in same scope → `Box<T>` or owned `Vec<T>`
- **BorrowShared**: read-only, not freed → `&T` or `&[T]`
- **BorrowMut**: written through, not freed → `&mut T` or `&mut [T]`
- **SharedOwnership**: stored in multiple places, may be freed from multiple locations → `Rc<T>` or `Arc<T>`
- **RawUnsafe**: cannot determine statically → requires human review or `unsafe` with justification

## Context for `rust` Paper

### Idiomatic Safe Rust for Streaming Parser Libraries

#### Rust's Ownership Model (Theoretical Foundation)

Rust's type system encodes ownership through three mechanisms:
1. **Ownership**: every value has exactly one owner; owner is responsible for deallocation (drop)
2. **Borrowing**: references `&T` (shared) and `&mut T` (exclusive) with lifetime parameters
3. **Lifetimes**: compile-time regions during which a borrow is valid; annotated as `'a`, `'b`, etc.

The borrow checker statically enforces: at any point, a value has either any number of shared references OR exactly one mutable reference, never both. This is the aliasing XOR mutability discipline that C violates freely.

#### Streaming Parser Design in Rust

A streaming parser for YAML (or XML, JSON) has a canonical Rust shape:

```rust
pub struct Parser {
    // internal state — owned
    reader: Box<dyn Read>,       // or a generic parameter
    buffer: Vec<u8>,             // input buffer (STRING equiv)
    token_queue: VecDeque<Token>, // QUEUE equiv
    state_stack: Vec<ParserState>, // STACK equiv
    indent_levels: Vec<i32>,
    marks: Vec<Mark>,
}

impl Parser {
    pub fn new() -> Self { ... }
    pub fn set_input_string(&mut self, input: &[u8]) { ... }
    pub fn set_input<R: Read + 'static>(&mut self, reader: R) { ... }
    
    #[must_use]
    pub fn parse(&mut self) -> Result<Event, Error> { ... }
    
    pub fn load(&mut self) -> Result<Document, Error> { ... }
}
```

The key design choices:
- **No output parameters**: `parse()` returns `Result<Event, Error>` rather than taking `*mut yaml_event_t`
- **Owned internal state**: buffer, queue, and stacks are owned by the Parser struct
- **`#[must_use]`** on Result-returning functions forces callers to handle errors
- **Generic input**: `impl Read` or `Box<dyn Read>` replaces C's function pointer + void* context

#### The libyaml-safer Lifetime Breakthrough

The critical insight from simonask's libyaml-safer port (published to crates.io February 9, 2024) was the Analysis struct lifetime problem:

In C, `yaml_emitter_analyze_event` stores pointers into the event being processed:
```c
// C: analysis pointers point into the event union
// yaml_emitter_t holds yaml_scalar_analysis_t with raw char* pointers
// into event data — the emitter is self-referential in C
```

A naive Rust port would make `yaml_emitter_t` self-referential (references to its own data), which is impossible without `unsafe` + raw pointers or `Pin`. The solution:

```rust
// Rust: Analysis<'a> borrows from the Event with lifetime 'a
struct Analysis<'a> {
    value: &'a str,  // borrows from the yaml_event_t
    // ...
}

// Analysis is passed as a parameter to functions that need it
fn emit_scalar<'a>(
    &mut self, 
    event: &'a Event, 
    analysis: &Analysis<'a>
) -> Result<(), Error> { ... }
```

The Analysis struct carries the lifetime `'a` from the event it analyzes, and is threaded through emitter functions as an extra parameter. This is forced by Rust's borrow checker, but it produces a *better design* than the C original: the `yaml_emitter_analyze_event` signature now explicitly documents that Analysis borrows from a yaml_event_t. The Rust compiler nudges toward better architecture.

**Key finding from simonask:** The port to safe, idiomatic Rust does not produce meaningfully slower code. Performance is largely on par with unsafe-libyaml.

#### Error Handling Strategy

**thiserror vs. anyhow tradeoff:**
- `thiserror` is for library crates that define their own Error type — callers can pattern-match on variants. Appropriate for `safe-libyaml` (a library).
- `anyhow` is for application code that just needs to propagate errors with context. NOT appropriate for a library crate.
- `safe-libyaml` should define a concrete `Error` type with `ErrorKind` enum (matching the C original's error categories), derived via `thiserror`.

```rust
use thiserror::Error;

#[derive(Debug, Error)]
pub enum Error {
    #[error("reader error at line {line}, column {column}: {problem}")]
    Reader { problem: String, line: u64, column: u64 },
    
    #[error("scanner error at line {line}, column {column}: {problem}")]
    Scanner { problem: String, line: u64, column: u64 },
    
    #[error("parser error at line {line}, column {column}: {problem}")]
    Parser { problem: String, line: u64, column: u64 },
    
    #[error("emitter error: {problem}")]
    Emitter { problem: String },
    
    #[error("memory error")]
    Memory,
}
```

#### no_std and alloc Considerations

For embedded targets, `safe-libyaml` should support `#![no_std]` with `extern crate alloc`:
- `Vec<T>` is available via `alloc::vec::Vec`
- `VecDeque<T>` is available via `alloc::collections::VecDeque`
- `String` is available via `alloc::string::String`
- `Box<dyn Read>` requires `alloc::boxed::Box` — but `dyn Trait` requires heap allocation
- For true no_std (no heap), a static buffer variant of the parser would be needed; this is a separate API concern

The primary `safe-libyaml` crate should use `std` by default with an optional `no_std` feature flag that switches to `alloc`.

#### FFI/ABI Exports via cbindgen

For C-compatible usage, safe-libyaml may need a C ABI wrapper:
```rust
// ffi.rs — C-compatible API layer

#[repr(C)]
pub struct SafeParser {
    inner: Box<crate::Parser>,
}

#[no_mangle]
pub extern "C" fn safe_libyaml_parser_new() -> *mut SafeParser {
    Box::into_raw(Box::new(SafeParser {
        inner: Box::new(crate::Parser::new()),
    }))
}

#[no_mangle]
pub unsafe extern "C" fn safe_libyaml_parser_free(p: *mut SafeParser) {
    if !p.is_null() {
        drop(Box::from_raw(p));
    }
}
```

cbindgen generates the corresponding C header automatically. The `#[repr(C)]` attribute ensures memory layout compatibility. The `unsafe` in `extern "C"` functions is unavoidable — the safety contract is that the caller maintains raw pointer validity.

#### The `unsafe` Boundary Discipline

For safe-libyaml specifically:
- **Zero `unsafe` in the core implementation** — enforced by CI (the unsafe-audit GitHub Actions job)
- **`unsafe` only in the FFI wrapper layer** with `// SAFETY:` comments explaining the invariant
- Miri must pass with zero errors on the entire test suite
- Any PR that adds an `unsafe` block requires explicit justification in the PR description

## Context for `c-rust-transpiling` Paper

### Automated C→Rust Transpiling: Pipeline, Patterns, and Verification

#### The c2rust Toolchain

c2rust (Immunant/Galois, DARPA-funded) is the foundational automated C→Rust transpiler. It operates at the AST/Clang level:
1. Parse C with Clang, extract the full AST
2. Map every C construct to its Rust equivalent
3. Emit `unsafe` Rust that mechanically mirrors the C

**c2rust produces unsafe Rust** — this is the starting point, not the endpoint. The unsafe-libyaml crate (github.com/dtolnay/unsafe-libyaml) is the exemplar: it is libyaml run through c2rust with post-processing cleanup, and it remains pervasively `unsafe`.

**c2rust v0.21 (October 2025) key changes:**
- Improved C preprocessor handling (freeform token-substitution)
- Portable types: recognizes `size_t`, `int32_t`, etc. and maps to `usize`, `i32`
- Compatible with LLVM 18–21, CMake 4
- Translates inline assembly to Rust inline assembly
- **Dropped c2rust-refactor tool**: the approach wasn't right for deep safety lifting; work started on a successor with more analysis capability
- New work on analysis to automate lifting unsafe Rust into safe Rust types

#### The Safety Ladder: C → Unsafe Rust → Safe Rust

The Ferrous Bridge pipeline explicitly conceptualizes a "ladder" of safety:
```
Rung 0: C source         (github.com/yaml/libyaml)
Rung 1: Unsafe Rust      (dtolnay/unsafe-libyaml — c2rust output)
Rung 2: Manual safe Rust (simonask/libyaml-safer — 1-week human effort)
Rung 3: AI safe Rust     (safe-libyaml — Ferrous Bridge target)
```

The AI-driven system uses Rung 1 as a mechanical reference and Rung 2 as the quality target. The goal is to produce output at Rung 2 quality automatically, then improve beyond it.

**Why unsafe-libyaml is still unsafe:**
- Raw pointer arithmetic preserved from C
- Direct memory operations (memcpy, memmove equivalents)
- Pointer casting between types
- Manual null checks (Option not used)
- C-style error code returns (not Result)
- Tagged union accessed via raw field access (not enum matching)
- The self-referential emitter analysis problem not solved

**What libyaml-safer did to climb to Rung 2:**
- Replaced all pointer operations with slice indexing
- Replaced all tagged unions with Rust enums
- Replaced C error returns with Result<T, Error>
- Replaced NULL checks with Option<T>
- Solved the Analysis<'a> lifetime problem (key insight for emitter)
- Used std::io::{Read, Write} instead of function pointer callbacks
- Denied unsafe code at crate level (#![deny(unsafe_code)])
- Achieved parity with unsafe-libyaml performance

#### Refactoring Patterns for the Safety Ladder

The key refactoring patterns that take c2rust output from unsafe to safe Rust:

| Unsafe pattern | Safe replacement |
|---|---|
| `*ptr` dereference | Slice indexing `buf[i]` or `buf.first()` |
| `ptr as *mut T` cast | Proper enum variant or newtype |
| `if ptr.is_null()` | `Option<T>` with `?` or `map()` |
| `return 0; // error` | `return Err(Error::...)?` |
| `memcpy(dst, src, n)` | `dst.copy_from_slice(&src[..n])` |
| `strcmp(a, b) == 0` | `a == b` |
| `strlen(s)` | `s.len()` |
| `malloc(size)` | `Vec::with_capacity(size)` |
| `free(ptr)` | Drop (automatic) |
| `struct { tag; union { ... } }` | `enum` |
| `void* context` callback | Closure / trait object |
| `(ptr, len)` pair | `&[T]` slice |

## Context for Synthesis Paper

### The Ferrous Bridge Pipeline: How the Three Compose

#### The Core Thesis

The Compile-Time Supremacy principle (c2rust_strategy.tex) unifies the three technical papers: C→Rust migration is not a syntactic translation problem but a *type inference* problem. The economic value created is proportional to the number of implicit runtime invariants successfully promoted to compile-time enforcement. A translation that produces `unsafe` Rust creates zero security value.

The three papers therefore compose as follows:
- **`c` paper**: characterizes the *source* — what implicit invariants C code encodes that must be recovered
- **`rust` paper**: characterizes the *target* — what explicit invariants safe Rust enforces and how to encode them idiomatically
- **`c-rust-transpiling` paper**: characterizes the *process* — how to recover those invariants automatically, verify the recovery, and iterate

#### The Five-Stage Pipeline Architecture

```
Stage 1: C Frontend Analysis (Open Source)
    Input: C source files
    Output: C-Analysis Bundle (CAB)
    Components: Clang AST, LLVM IR, points-to analysis (Andersen-style),
                call graph, use-def chains, dynamic traces (ASan/MSan/TSan)
    Key crate: fb-frontend

Stage 2: Ownership Semantic Graph (Open Core + Proprietary)
    Input: CAB
    Output: OSG — annotated functions with ownership class, Rust idiom, confidence
    Components: ownership classifier (heuristic → ML), lifetime region analysis,
                allocator pattern recognition, idiom detection
    Key crate: fb-osg

Stage 3: Translation Engine (Proprietary Core)
    Input: OSG + C source
    Output: Rust code candidates
    Components: router (FTM vs Claude), Claude API integration,
                fine-tuned model (DeepSeek/Qwen), prompt templates
    Key crate: fb-translate

Stage 4: Verification Pipeline (Open Core + Proprietary)
    Input: C source + Rust candidate
    Output: pass/fail verdict with structured error feedback
    Components: rustc gate, Miri gate, differential fuzz gate (10K inputs),
                Clippy gate, complexity gate; KaniGate + CreusotGate in high_assurance()
    Key crate: fb-verify

Stage 5: Human Review Platform (Proprietary)
    Input: verified Rust + confidence scores
    Output: approved translation + training signal
    Components: Axum HTTP API, React frontend, side-by-side diff,
                annotation tools, audit trail, feedback loop
    Key crate: fb-review
```

**Feedback loops:**
- Stage 4 rejection → Stage 3 (re-translate with error context)
- Stage 5 annotation → Stage 2 (retrain ownership classifier)
- Stage 5 approval → Training Data Pipeline (new verified pair)

## Topic Overviews

### libyaml Internals

**Background:** libyaml is a C library implementing YAML 1.1 parsing and emitting. Originally written by Kirill Simonov. Maintained by the YAML community. Released under MIT license. It provides a streaming event-based API (like SAX for XML) rather than building a full document tree.

**Parser/emitter architecture:** Two processors: (1) Parser takes input stream of bytes → sequence of parsing events; (2) Emitter takes sequence of events → stream of bytes. The loader/dumper layer adds document tree construction on top.

**yaml_event_t:** The central tagged union. The `type` field is the discriminant:
- YAML_STREAM_START_EVENT / YAML_STREAM_END_EVENT
- YAML_DOCUMENT_START_EVENT / YAML_DOCUMENT_END_EVENT
- YAML_ALIAS_EVENT
- YAML_SCALAR_EVENT (most complex: value, length, anchor, tag, style)
- YAML_SEQUENCE_START_EVENT / YAML_SEQUENCE_END_EVENT
- YAML_MAPPING_START_EVENT / YAML_MAPPING_END_EVENT

**Custom allocator hooks:** yaml_parser_t and yaml_emitter_t contain function pointer fields `yaml_malloc_t`, `yaml_realloc_t`, `yaml_free_t` plus a `void* data` context. This allows callers to provide custom memory allocators. In safe Rust, these become either: the global allocator (Vec/Box), a bumpalo arena for performance-critical paths, or a user-provided allocator implementing the Allocator trait.

**STACK_INIT, STACK_PUSH, STACK_POP mechanics:**
- INITIAL_STACK_SIZE = 16 elements
- PUSH: if `stack.top == stack.end`, call yaml_stack_extend (doubles capacity via realloc)
- POP: decrement top pointer, return value
- DEL: free memory, zero pointers

**QUEUE_INIT, ENQUEUE, DEQUEUE mechanics:**
- INITIAL_QUEUE_SIZE = 16 elements
- ENQUEUE: if `queue.tail == queue.end`, call yaml_queue_extend (may wrap around or realloc)
- DEQUEUE: advance head pointer
- DEL: free memory, zero pointers

**scanner.c key complexity:** The YAML scanner is a hand-written lexer with complex lookahead (CHECK, CHECK_AT macros that look N bytes ahead), Unicode handling (IS_BOM, IS_BLANKZ, IS_BREAKZ predicates), block/flow context tracking, and a simple_key stack for implicit key detection in block mappings. The IS_BLANK, IS_BREAK, IS_BREAKZ, IS_BLANKZ, IS_Z (zero) predicates define the character classes.

### libexpat Internals

**Background:** Expat (libexpat) is a stream-oriented XML parser written in C. ~20K LOC. Used by Python (xml.parsers.expat), Apache, Firefox, D-Bus, many others. Originally by James Clark. Memory-unsafe implementation with extensive CVE history.

**XML_Parser API:** Central opaque struct `XML_Parser`. Key functions:
- `XML_ParserCreate(encoding)` → XML_Parser
- `XML_SetUserData(parser, userData)` — void* context for callbacks
- `XML_SetElementHandler(parser, startElement, endElement)` — SAX-style callbacks
- `XML_Parse(parser, buffer, len, isFinal)` → XML_Status
- `XML_ParserFree(parser)`

**Memory handlers:**
```c
typedef void* (*XML_MallocFunction)(size_t size);
typedef void* (*XML_ReallocFunction)(void *ptr, size_t size);
typedef void (*XML_FreeFunction)(void *ptr);

typedef struct {
    XML_MallocFunction malloc_fcn;
    XML_ReallocFunction realloc_fcn;
    XML_FreeFunction free_fcn;
} XML_Memory_Handling_Suite;
```

**CVE history (selected):**
- CVE-2022-25235: Missing UTF-8 validation in xmltok_impl.c; allows passing malformed 2/3-byte sequences; exploits with code execution known
- CVE-2022-25236: Namespace separator injection allowing XML_ExternalEntityParserCreate abuse
- CVE-2024-8176: Stack overflow via recursive entity expansion; deeply nested XML causes stack exhaustion
- CVE-2025-59375: Dynamic memory allocation vulnerability; large allocations via small documents
- 40+ total CVEs since library creation; continuous stream of memory safety bugs

**Translation challenges:** SAX-style callbacks (void* context → closures), DTD processing (recursive structure), encoding handling (UTF-8/16/32/Latin-1 similar to libyaml reader.c), XML_StopParser for abort-on-error (→ early return from iterator), entity references and expansion (recursion depth limits needed in Rust).

### Academic C→Rust Translation Landscape

**The core problem statement:** Approximately 100x more C code in public repositories than Rust. Aligned C↔Rust translation pairs are extremely scarce. The bottleneck for fine-tuning is verified training data.

**Fundamental approaches:**
1. **AST-level transpilation** (c2rust, Immunant): mechanical, always produces unsafe output; well-defined but produces the wrong thing
2. **Ownership analysis + refactoring** (Crown, Laertes): static analysis to infer ownership → automated refactoring; scalable but incomplete
3. **LLM translation** (SACTOR, RustMap, Syzygy, Rustine): uses LLM to generate idiomatic Rust; higher quality but needs verification
4. **Formal methods + LLM** (ForCLift): combines verified lifting with LLM for hard cases; highest correctness guarantee; research stage
5. **Hybrid pipelines** (Ferrous Bridge): static analysis (ownership inference) → LLM (translation) → formal verification (correctness) → human review (training signal)

**Key open problems:**
- Cross-function lifetime inference across module boundaries (context window limitation)
- Custom allocator migration (arena, pool, slab → bumpalo, typed-arena)
- Concurrent code translation (pthread → Arc<Mutex<T>>, Loom verification)
- setjmp/longjmp non-local control flow (libpng)
- Verified semantic equivalence at scale (symbolic execution impractical; differential fuzzing scales but doesn't prove)
- The "intent vs. behavior" problem: translated Rust must not reproduce bugs present in C
